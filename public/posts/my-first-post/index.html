A Deep Dive into EVM Gas Optimization

July 6, 2025 4-minute read
Solidity • Gas Optimization • EVM • Blockchain

Optimizing for gas is a critical skill for any Solidity developer. Lower gas costs mean cheaper transactions for users and more efficient applications. Here’s a breakdown of the main areas to focus on for gas optimization.

1. Storage Link to heading

Storage is the most expensive resource in the EVM.

  • Initial write: Saving one variable to a new storage slot costs 20,000 gas.
  • Rewrite: Rewriting an existing variable costs 5,000 gas.
  • Read: Reading from a storage slot costs 200 gas.

Tips:

  • Use Memory Caching: Always load storage variables into memory within a function before using them multiple times. If you need to update a storage variable, perform all calculations on memory variables first and then write the final result back to storage once.
  • Pack Your Variables: Try to organize and pack two or more smaller storage variables into a single 32-byte slot. This is significantly cheaper. This also applies when designing structs.
  • Refunds:
    • Free up a storage slot by setting its value to zero (0) as soon as it’s no longer needed. This refunds 15,000 gas.
    • Destroying a contract using selfdestruct refunds 24,000 gas. Note that refunds cannot exceed half the gas used by the current transaction.

2. Data Types and Packing Link to heading

  • bytes32 is the most optimized storage type. Use it whenever possible. If you need a smaller bytes type, use the smallest one that fits your data (e.g., bytes1 to bytes31).
  • bytes32 is cheaper than string or bytes.
  • Important: Variable packing only occurs in storage. It does not happen for function arguments or local variables in memory. You won’t save space trying to pack these.
  • Storing a small number in uint8 isn’t cheaper than uint256 in memory or calldata, as it gets padded to fill 32 bytes anyway. Packing only matters for storage.

3. Functions Link to heading

  • Visibility: Use external visibility whenever possible. It’s cheaper than public because the function parameters don’t need to be copied to memory.
  • Ordering: The EVM uses a function selector to identify which function to call. Put frequently called functions earlier in the contract to potentially save a tiny amount of gas (average of 22 gas per position).
  • payable Modifier: A payable function is slightly cheaper than a non-payable one because the compiler doesn’t need to add a check to reject incoming Ether.
  • View Functions: While view functions are free to call from off-chain, they do consume gas when called from another function within a transaction.

4. Loops Link to heading

  • Cache array lengths and other values in memory variables before starting a loop.
  • Avoid unbounded loops that could potentially run forever or exceed the block gas limit.
  • Instead of uint256 i = 0;, just use uint256 i;. It will default to zero and saves a small amount of gas on initialization.
  • Use prefix increment (++i) instead of postfix increment (i++). It’s slightly cheaper.

5. Operations Link to heading

  • Short-Circuiting: Structure your require statements or conditional logic to take advantage of short-circuiting rules.

    • For f(x) || g(y) (OR), place the cheaper or more likely true function first.
    • For f(x) && g(y) (AND), place the cheaper or more likely false function first.

  • unchecked Blocks: Since Solidity v0.8.0, all arithmetic operations are checked for overflow/underflow. If you are certain an operation will not over/underflow, you can wrap it in an unchecked block to save gas.

    // Example
unchecked {
    counter++;
}

6. Other Optimizations Link to heading

  • Use keccak256: It’s the cheapest hash function available in Solidity (30 gas + 6 gas per word).
  • Use Minimal Proxies (ERC-1167): If you need to deploy many instances of the same contract, use the ERC-1167 standard. It creates cheap, immutable clones of a base implementation contract.
  • Merkle Proofs: To validate a large set of data on-chain, use a Merkle tree. You can store only the Merkle root on-chain and have users provide a Merkle proof to validate their data, which is very gas-efficient.
  • Custom Errors: Instead of using long require strings, define custom errors. They are significantly cheaper starting from Solidity v0.8.4. 
    error NotTheOwner();
// ...
if (msg.sender != owner) {
    revert NotTheOwner();
}

Tools for Estimating Gas Link to heading

  • Remix IDE: Provides gas estimates directly in the editor.
  • Truffle: Can be configured to report gas usage for tests.
  • hardhat-gas-reporter: An excellent Hardhat plugin that reports gas usage for your test suite, helping you track optimizations.
